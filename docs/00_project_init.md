# 云边端协同智能巡检与运维平台：项目立项说明书

## 1. 项目背景

随着智慧机房、实验室、小型数据中心和工业现场设备数量不断增加，传统人工巡检方式存在效率低、实时性差、问题发现滞后、故障处理依赖个人经验等问题。

本项目希望构建一套云边端协同的智能巡检与运维平台，通过设备数据采集、边缘网关接入、智能告警、知识库问答、ROS2巡检机器人和AI视觉识别等能力，实现从设备状态感知、异常发现、故障分析到处理建议生成的完整闭环。

## 2. 项目建设意义

本项目的建设意义包括：

1. 提升机房和实验室设备巡检效率；
2. 降低人工巡检遗漏风险；
3. 实现设备状态的实时监控；
4. 通过告警规则及时发现异常；
5. 通过知识库和智能助手辅助运维人员排查问题；
6. 为后续接入ROS2机器人和AI视觉巡检打下基础；
7. 通过完整项目流程训练公司级项目开发能力。

## 3. 目标用户

本项目的目标用户包括：

- 系统管理员
- 运维工程师
- 实验室管理员
- 机房管理人员
- 设备维护人员
- 项目管理人员

## 4. 典型使用场景

### 4.1 设备状态监控

运维人员登录平台后，可以查看所有设备的在线状态、运行指标和最近上报时间。

### 4.2 异常告警处理

当设备CPU、内存、磁盘、温度、湿度等指标超过阈值时，系统自动生成告警记录。

### 4.3 巡检任务管理

后续版本中，系统可以创建巡检任务，并下发给边缘网关或ROS2巡检机器人。

### 4.4 运维知识问答

后续版本中，运维人员可以通过智能助手查询故障处理方法，例如“服务器磁盘满了怎么办”。

### 4.5 AI视觉巡检

后续版本中，系统可以通过摄像头或机器人采集图像，识别仪表、指示灯、设备异常等内容。

## 5. 项目目标

本项目分阶段完成：

- V1：完成基础平台，包括用户、设备、数据上报、监控和告警；
- V2：完成边缘网关和MQTT设备接入；
- V3：完成RAG运维助手；
- V4：完成ROS2巡检机器人接入；
- V5：完成AI视觉巡检、监控体系和工程化部署。

## 8. 技术路线

本项目采用：

- FastAPI 作为后端框架；
- React + TypeScript 作为前端框架；
- PostgreSQL 作为业务数据库；
- Redis 作为缓存和设备状态存储；
- Docker Compose 作为初期部署方式；
- MQTT 作为后续设备通信协议；
- ROS2 作为后续机器人接入技术；
- OpenCV / YOLO 作为后续AI视觉巡检能力；
- RAG 作为后续运维知识问答能力。

## 9. 项目风险

| 风险 | 说明 | 应对策略 |
|---|---|---|
| 项目范围过大 | 云、边、端、AI、ROS2 都涉及，容易失控 | 分版本迭代，先完成V1 |
| 硬件调试复杂 | 真实设备接入可能耗时较长 | 前期使用模拟设备 |
| AI效果不稳定 | 视觉识别依赖数据质量 | 先做简单可控场景 |
| 工程复杂度高 | 模块较多，容易混乱 | 坚持文档、分支、版本管理 |
| 部署难度增加 | 后期涉及多服务 | 先Docker Compose，后K8s |
